# 项目架构说明

版本：v0.1

日期：2026-03-26

## 1. 架构结论

本项目首版采用：

- 单仓库
- 单个 `Next.js` 应用
- 前台、用户中心、管理后台、API 一体化承载
- 代码层面严格分层，不混写业务逻辑

## 2. 为什么不拆成前后端两个项目

当前项目是社区 MVP，首版重点是尽快跑通以下闭环：

- 注册审核登录
- 圈子浏览与圈子申请
- 发帖、评论、投票、附件
- 举报、敏感词、新用户审核
- 后台治理与轻运营

在这个阶段拆成两个独立项目，会增加：

- 工程初始化成本
- 联调成本
- 登录态和权限共享成本
- 后台与前台之间的接口维护成本

因此首版更适合放在一个工程内完成。

## 3. 一体化不等于混在一起

虽然项目工程放在一起，但职责必须分清：

- `src/app/(site)`：公开前台页面
- `src/app/(user)`：用户中心页面
- `src/app/admin`：管理后台页面
- `src/app/api`：接口入口层
- `src/server`：服务端配置与后续业务服务层
- `src/modules`：按业务模块拆分能力
- `src/components`：可复用组件
- `src/lib`：前端共享工具与静态数据

## 4. 推荐开发方式

建议开发时遵守以下顺序：

1. 先建数据模型
2. 再建服务层和权限边界
3. 再补 API
4. 最后落页面交互

不要把数据库查询、权限判断、页面渲染全部直接写进同一个页面文件。

## 5. 什么时候再考虑拆分

后续如果出现以下情况，可以再评估拆分独立后端服务：

- 需要同时支撑 App、小程序、开放 API
- 实时消息、审核流水线、任务队列明显变重
- 团队规模扩大到需要前后端独立部署
- 部署架构从单机演进到多服务协作

在此之前，保持单仓一体化更适合当前阶段。
